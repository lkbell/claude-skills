# Tasks — run Landon's ClickUp board

Landon's to-dos live on a ClickUp kanban board that **the assistant operates and Landon supervises** — the assistant does the hands-on work; Landon reviews and redirects on the board, and his own edits there take precedence.

Before working the board, **read the ✅ Tasks operating model** and follow it. It lives in **`tasks/README.md`** in the repo **lkbell/brain**, via **GitHub**. The model holds what you can't guess: the Lists and how to route to them, the status set, task formatting, where tasks come from, the board-vs-docs split, the cadence, and the guardrails. (This skill points at `tasks/README.md` and hardcodes nothing else, so it survives brain restructuring.)

Hold these rules up front:

- Use ClickUp as the working file for the task. Include concise information genuinely useful for executing, following up on, or substantiating it, including task-specific contacts, reference numbers, and necessary context. Do not leave important knowledge there if Landon would reasonably want it after the task closes or outside that task; file that knowledge in the Brain and link it. Summarize source material instead of copying records or narrative history. Do not treat an existing detailed card as precedent.
- After creating or materially updating a task, fetch or reopen the live card. Before reporting completion, correct it if it has become a repository for source material or important life knowledge rather than a useful working task.
- Capture only in a live session (a scheduled job cannot see the conversation). Act within the Bible's guardrails: make reversible board changes freely, but confirm before anything that reaches a third party (for example, a calendar invitation), spends money, or loses data.

Requires ClickUp access (the board itself) and GitHub access (`tasks/README.md` in `lkbell/brain`, for the operating model). If either isn't available on this surface, say so and proceed without it.
